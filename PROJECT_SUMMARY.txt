1. PROJECT OVERVIEW
================================================================================

The AI HR Assistant is a production-ready, full-stack SaaS application that
allows employees to ask questions about company HR policies and get instant,
AI-powered answers grounded in actual company documents using Retrieval-
Augmented Generation (RAG).

Instead of employees browsing through long policy PDFs or waiting days for HR
to reply to emails, they can simply chat with the AI assistant. The assistant
reads company HR documents (leave policy, office hours, project management
guidelines), finds the most relevant sections, and generates accurate answers
with source citations and confidence scores.

An HR admin dashboard provides real-time analytics — tracking total questions,
employee activity, AI accuracy, feedback trends, and conversation logs — so HR
managers can monitor and improve the system.

2. PROBLEM STATEMENT & SOLUTION
================================================================================

PROBLEM:
  - Employees waste time searching through lengthy HR policy documents
  - HR teams repeatedly answer the same questions (leave rules, office hours,
    project management procedures)
  - No centralized, intelligent system to handle HR queries at scale
  - No analytics on what employees are asking or how accurate the answers are

SOLUTION:
  - AI chatbot that answers HR questions instantly using RAG (Retrieval-
    Augmented Generation) — answers are grounded in real company documents,
    not hallucinated
  - Every answer includes source citations (which document, which section) and
    a confidence score so employees know how reliable the answer is
  - HR admin dashboard with real-time analytics, conversation logs, feedback
    management, and knowledge base document management
  - Streaming responses (token-by-token) for a ChatGPT-like experience

4. ARCHITECTURE & DATA FLOW
================================================================================

HIGH-LEVEL ARCHITECTURE:

  [Employee Browser]
        |
        v
  [Next.js Frontend (port 3000)]
        |
        |--- /chat          -> Chat interface (employee-facing)
        |--- /login         -> Supabase Auth login
        |--- /dashboard/*   -> HR Admin dashboard (protected)
        |
        v
  [FastAPI Backend (port 8000)]
        |
        |--- /chat          -> RAG pipeline -> OpenRouter LLM
        |--- /chat/stream   -> Streaming SSE response
        |--- /feedback      -> Save to Supabase
        |--- /stats         -> Analytics engine
        |--- /analytics     -> Detailed analytics
        |--- /logs          -> Conversation history
        |--- /documents     -> Knowledge base management
        |--- /health        -> Health check
        |
        +--- [ChromaDB Vector Store]  (local, persisted)
        +--- [Supabase PostgreSQL]    (cloud database)
        +--- [OpenRouter API]         (LLM inference)

RAG PIPELINE FLOW (How AI answers are generated):

  Step 1: DOCUMENT LOADING
    - HR policy files (.md, .json, .txt) are loaded from /documents folder
    - Markdown files are split by headers (## sections)
    - JSON files are flattened into key-value pairs
    - Each chunk gets metadata: source file, section name, file type

  Step 2: CHUNKING
    - Documents split into ~1000 character chunks with 200 char overlap
    - Uses LangChain's RecursiveCharacterTextSplitter
    - Separators: paragraph -> newline -> sentence -> word -> character

  Step 3: EMBEDDING
    - Each chunk is converted to a 384-dim vector using all-MiniLM-L6-v2
    - Embeddings run locally (no API call needed)
    - SentenceTransformerEmbeddings wraps the model for LangChain compatibility

  Step 4: VECTOR STORAGE
    - Embeddings stored in ChromaDB (persisted to ./vectorstore directory)
    - Supports cosine similarity search
    - Vector store can be rebuilt via POST /documents/rebuild endpoint

  Step 5: RETRIEVAL (when employee asks a question)
    - Question is embedded using the same model
    - ChromaDB finds top-5 most similar chunks (cosine similarity)
    - Chunks with relevance score < 0.1 are filtered out
    - Retrieved chunks become the "context" for the LLM

  Step 6: GENERATION
    - System prompt + retrieved context + user question sent to OpenRouter
    - LLM generates answer using ONLY the provided context (no hallucination)
    - Temperature: 0.3 (low creativity, high accuracy)
    - Max tokens: 2000

  Step 7: CITATION & CONFIDENCE
    - Source documents (name, section, relevance score) returned with answer
    - Confidence score = average relevance score * 1.2 (capped at 1.0)
    - Response time measured in milliseconds

STREAMING FLOW (for /chat/stream endpoint):
    - Server-Sent Events (SSE) used for real-time token streaming
    - First event: metadata (chat_id, confidence, sources, retrieval time)
    - Subsequent events: content chunks (token-by-token from LLM)
    - Final event: done (with chat_id for feedback submission)
    - Frontend uses Fetch API with ReadableStream to parse SSE events

5. DATABASE SCHEMA (SUPABASE / POSTGRESQL)
================================================================================

TABLES:

  1. admin_users
     - id (UUID, PK)
     - email (TEXT, UNIQUE)
     - full_name (TEXT)
     - role (TEXT: admin / super_admin / hr_manager)
     - created_at, updated_at (TIMESTAMPTZ)

  2. employees
     - id (UUID, PK)
     - name (TEXT)
     - employee_id (TEXT, UNIQUE)
     - department, designation (TEXT)
     - is_active (BOOLEAN)
     - created_at, updated_at (TIMESTAMPTZ)

  3. hr_documents
     - id (UUID, PK)
     - name, type, content, file_path (TEXT)
     - sections (INT)
     - uploaded_by (UUID -> admin_users)
     - created_at, updated_at (TIMESTAMPTZ)

  4. chat_history (MAIN TABLE)
     - id (UUID, PK)
     - employee_name, employee_id (TEXT)
     - question (TEXT)
     - ai_response (TEXT)
     - retrieved_context (TEXT)
     - source_documents (JSONB)
     - confidence_score (FLOAT)
     - response_time_ms (INT)
     - feedback (TEXT: positive / negative / NULL)
     - feedback_notes (TEXT)
     - flagged (BOOLEAN)
     - session_id (UUID)
     - created_at, updated_at (TIMESTAMPTZ)

  5. ai_feedback
     - id (UUID, PK)
     - chat_id (UUID -> chat_history, CASCADE)
     - feedback_type (TEXT: positive / negative)
     - notes, given_by (TEXT)
     - created_at (TIMESTAMPTZ)

  6. analytics
     - id (UUID, PK)
     - metric_name (TEXT)
     - metric_value (FLOAT)
     - metric_date (DATE)

  7. user_sessions
     - id (UUID, PK)
     - employee_id (TEXT)
     - session data
     - created_at, updated_at (TIMESTAMPTZ)

SECURITY:
  - Row Level Security (RLS) enabled on ALL tables
  - Authenticated users can SELECT/INSERT on chat_history
  - Service role key bypasses RLS (used by backend only)
  - Auto-trigger: feedback changes logged to ai_feedback table
  - Auto-trigger: updated_at column auto-updated on row changes

6. API ENDPOINTS
================================================================================

  METHOD  PATH                  DESCRIPTION
  ------  ----                  -----------
  POST    /chat                 Send question, get complete AI response
  POST    /chat/stream          Send question, get streaming SSE response
  POST    /feedback             Submit feedback (positive/negative/flagged)
  GET     /stats                Summary statistics (totals, averages)
  GET     /analytics            Detailed analytics (charts, topics, employees)
  GET     /logs                 Conversation logs (paginated, searchable)
  GET     /documents            List knowledge base documents
  POST    /documents/rebuild    Rebuild RAG vector store from documents
  GET     /health               Health check (service status)
  GET     /                     API info (name, version, docs URL)
  GET     /docs                 Interactive Swagger UI documentation

REQUEST/RESPONSE MODELS:

  ChatRequest:
    - question (str, 1-2000 chars)
    - employee_name (str, 1-200 chars)
    - employee_id (str, 1-100 chars)
    - session_id (str, optional)

  ChatResponse:
    - id (str)
    - answer (str)
    - confidence (float)
    - sources (List[SourceDocument])
    - response_time_ms (int)

  FeedbackRequest:
    - chat_id (str)
    - feedback ("positive" | "negative")
    - flagged (bool)
    - notes (str, optional)

10. INSTALLATION & SETUP GUIDE
================================================================================

PREREQUISITES:
  - Node.js 18+ and npm
  - Python 3.11+
  - Docker & Docker Compose (optional, for containerized deployment)
  - A Supabase project (free tier at https://supabase.com)
  - An OpenRouter API key (free at https://openrouter.ai)

STEP 1: CLONE THE REPOSITORY
  git clone https://github.com/Shazia-Amir/AI-HR-Assistant.git
  cd AI-HR-Assistant

STEP 2: ENVIRONMENT SETUP
  Copy the example env file and fill in your credentials:
    cp .env.example .env

  Required variables:
    # Supabase
    NEXT_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
    NEXT_PUBLIC_SUPABASE_ANON_KEY=your-anon-key
    SUPABASE_SERVICE_ROLE_KEY=your-service-role-key

    # OpenRouter
    OPENROUTER_API_KEY=your-openrouter-api-key
    OPENROUTER_MODEL=deepseek/deepseek-chat-v3-0324

    # Backend
    BACKEND_HOST=0.0.0.0
    BACKEND_PORT=8000
    CORS_ORIGINS=http://localhost:3000

    # RAG
    CHUNK_SIZE=1000
    CHUNK_OVERLAP=200
    TOP_K_RESULTS=5
    EMBEDDING_MODEL=all-MiniLM-L6-v2
    VECTOR_STORE_PATH=./vectorstore

STEP 3: DATABASE SETUP
  - Go to your Supabase project dashboard
  - Open the SQL Editor
  - Copy and paste the contents of supabase/schema.sql
  - Execute the query
  - This creates all tables, indexes, RLS policies, and triggers

STEP 4: BACKEND SETUP
  cd backend
  python -m venv venv

  # Activate virtual environment:
  #   Windows:  venv\Scripts\activate
  #   Mac/Linux: source venv/bin/activate

  pip install -r requirements.txt
  uvicorn main:app --reload --host 0.0.0.0 --port 8000

  Backend will be available at http://localhost:8000
  API docs at http://localhost:8000/docs

STEP 5: FRONTEND SETUP
  cd frontend
  npm install
  npm run dev

  Frontend will be available at http://localhost:3000

STEP 6: DOCKER (OPTIONAL — RUN BOTH WITH ONE COMMAND)
  docker-compose up --build

  This starts:
    - Backend container on port 8000
    - Frontend container on port 3000

================================================================================

  1. leave_policy.md / .json
     - Annual leave, sick leave, casual leave policies

13. SECURITY IMPLEMENTATION
================================================================================

  - Environment variables: All secrets stored in .env (gitignored)
  - Supabase RLS: Row Level Security on all database tables
  - Service role key: Used only by backend (bypasses RLS)
  - CORS: Configurable allowed origins (no wildcard with credentials)
  - Rate limiting: slowapi middleware (configurable per-minute limit)

  - Auth middleware: Next.js middleware protects /dashboard/* routes
  - Session management: Supabase SSR cookie-based sessions
  - Error handling: Global exception handler prevents info leakage

14. TROUBLESHOOTING
================================================================================

  PROBLEM                        SOLUTION
  -------                        --------
  RAG pipeline fails             Ensure /documents has .md or .json files.
                                 Pipeline defers init but API still works.

  ChromaDB errors                Delete /vectorstore folder and restart
                                 backend. It rebuilds on next request.

  Supabase connection refused    Check NEXT_PUBLIC_SUPABASE_URL and
                                 NEXT_PUBLIC_SUPABASE_ANON_KEY in .env.
                                 Ensure Supabase project is active.

  OpenRouter 401 Unauthorized    Verify OPENROUTER_API_KEY is valid.
                                 Get free key at openrouter.ai.

  CORS errors in browser         Add frontend URL to CORS_ORIGINS in .env.

  npm run dev fails              Delete node_modules/ and package-lock.json,
                                 then run npm install again.

  Python dependency conflicts    Create fresh venv: python -m venv venv,
                                 activate, and reinstall.

  Port already in use            Change BACKEND_PORT in .env or kill the
                                 process using that port.
